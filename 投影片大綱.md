## Slide 5: Antigravity 是什麼？

**定義：** 程式碼自動生成平台  
**價值：** 讓非工程師也能參與開發

**傳統 vs Antigravity**

| 項目 | 傳統 | Antigravity |
|------|------|-------------|
| 需求到程式碼 | 手寫，數天至數週 | 自然語言 → 自動生成，數分鐘 |
| 所需人力 | 前端 + 後端工程師 | 1 人或甚至不需 |
| 迭代速度 | 慢（每次改動都要重寫） | 快（調整描述即可重新生成） |

## Slide 10: GitHub Pages - 靜態網頁托管

**原理：** 把 HTML 推到 GitHub，它幫你host

**部署步驟：**
1. 準備 `index.html`（由 Antigravity 生成）
2. `git push` 到 GitHub 倉庫
3. Settings → Pages → 選擇分支 `/root`
4. 完成！網站上線

**優點：**
- 免費、HTTPS、自訂網域
- 自動部署（每次 push）

**限制：** 無法寫入資料庫（純靜態）

## Slide 12: Hugging Face Spaces - 部署 Gradio API

**什麼是 Spaces？**
- 原本是 AI 模型展示平台
- 支援 Gradio、Streamlit、Docker
- 免費 hosting（CPU，16GB RAM）

**部署步驟：**
1. 註冊 Hugging Face
2. New Space → 選 Gradio SDK
3. 上傳 `app.py` + `requirements.txt`
4. 自動建置 + 上線

---

## Slide 13: Hugging Face Spaces 技術細節

**自動偵測流程：**
```
上傳檔案 → 安裝 requirements.txt → 執行 app.py → 公開 URL
```

**`requirements.txt` 範例**
```txt
gradio>=4.0
folium
pandas
```

**注意事項：**
- 避免巨大套件（會拖慢 build）
- 不要放入 `__pycache__`
- 需要 token 時使用 Settings → Repository secrets

---

## Slide 14: Spaces 限制與最佳實踐

| 限制 | 數值 | 建議 |
|------|------|------|
| 記憶體 | ~16 GB | 勿載入大模型 |
| timeout | 幾分鐘 | 長時間 task 用 queue |
| 儲存 | 暫存盤（重啟消失） | 用外部 DB 或 HF Dataset |
| concurrent | 有限制 | 加排隊或升級 |

**最佳實踐：**
- 版本固定（`gradio==4.28.3`）
- 使用 `cache_examples=True` 快取結果
- 複雜推理改用 Inference API（非本地載入）

## Slide 18: config.yaml 核心配置

```yaml
connectors:
  telegram:
    bot_token: "${TELEGRAM_BOT_TOKEN}"
    allowed_user_ids: []

integrations:
  huggingface:
    api_url: "https://api-inference.huggingface.co/models/..."
    token: "${HF_TOKEN}"
  github:
    repo: "username/disaster-dashboard"
    token: "${GITHUB_TOKEN}"
    data_file: "disaster_data.json"

workflows:
  - id: "disaster_report"
    trigger: "telegram.message"
    steps:
      - parse_message
      - assess_risk
      - update_github
      - send_email
      - reply_user
```

---

## Slide 19: Workflow 工作流詳細步驟

**災害通報處理流程：**

1. **parse_message** - 解析 Telegram 訊息（地點、描述、圖片）
2. **assess_risk** - 呼叫 Hugging Face API 判斷風險等級
3. **format_data** - 組織成 JSON 格式
4. **update_github** - 更新 `disaster_data.json`
5. **send_email** - 若等級為「高」則發送警報 Email
6. **reply_telegram** - 回傳確認訊息給同仁

## Slide 27: 常見問題與回答（Q&A 準備）

**Q1: 資料安全嗎？會不會被別人看到？**
- GitHub Pages 可設為 Private repo，或搭配 token 限制 API
- Openclaw 可設定 `allowed_user_ids` 只允許特定人員
- 敏感性資料建議內部部署，不公開

**Q2: HF API 延遲高怎麼辦？**
- 使用 fallback rule-based 評估
- 設定 retry 機制
- 高優先案例用更高級方案（Pro）

**Q3: 坐標怎麼來？**
- 手機 GPS 直接傳緯度
- 地址轉坐標：Google Maps API / OpenStreetMap Nominatim

**Q4: 系統如何備份？**
- GitHub 本身就是版本控制
- 定期 export CSV 或存入外部 DB
